# Enzyme AI Papers Project Blueprint

This blueprint defines the URL-first version of the project.

## Project Goal

Enzyme AI Papers is a curated paper index and weekly digest for AI and
computational methods related to enzymes. The platform should let community
members recommend useful papers with minimal effort while keeping human
curation as the quality gate.

The core promise is:

```text
Submit a paper URL. Maintainers review a preview and apply labels. The site is generated automatically.
```

## Design Principles

1. Keep contribution friction low.
   A contributor should only need to paste a URL. Title, tags, and notes are
   useful but optional.

2. Keep maintainer decisions lightweight.
   Maintainers should decide whether to include, feature, request more context,
   or reject. They should not need to hand-write weekly files.

3. Keep accepted data reviewable.
   Accepted papers still live as YAML under `data/papers/`, but those records
   are generated by automation and reviewed through pull requests.

4. Keep weekly digests derived.
   Weekly issues are grouped by `accepted_at` using ISO week names such as
   `YYYY-Www`. These week names are internal generated output, not user input.

5. Keep automation best-effort.
   Metadata lookup can fail or be incomplete. The generated pull request remains
   reviewable and editable.

## User Workflow

1. A contributor uses the website Submit form or opens a paper suggestion issue.
2. The only required field is the paper URL.
3. The website opens a prefilled GitHub issue; it does not write repository data.
4. GitHub Actions comments with a metadata preview.
5. The contributor can add a note or suggested free-text tags if they want.

## Maintainer Workflow

Maintainers use labels:

- `accepted`: include the paper.
- `featured`: mark it as a Pick of the Week candidate.
- `needs-info`: request more context.
- `rejected`: decline.

When `accepted` is applied, the curation workflow:

1. Parses the issue.
2. Performs best-effort metadata lookup.
3. Infers canonical tags from title, note, and suggested tags.
4. Checks for existing accepted records.
5. Writes or updates `data/papers/YYYY/<paper-id>.yml`.
6. Regenerates `README.md` and `docs/`.
7. Opens a pull request for review.

## Data Model

Accepted papers live under `data/papers/YYYY/`. The stable record includes:

- bibliographic metadata: `title`, `authors`, `year`, `date`, `source`, `doi`
- links: `url`, `pdf`, `code`, `project`, `preprint_url`, `published_url`
- curation metadata: `accepted_at`, `issue`, `submitted_by`, `reviewed_by`,
  `featured`, `curation`
- taxonomy: `topics`, `methods`, `evidence`, `applications`
- readable summaries: `one_liner`, `why_it_matters`

`data/weekly/` is optional. A file there can override the generated weekly
title, summary, pick, sections, or commentary for a specific week. It is not
required for normal curation.

## Website

The generated MkDocs site has three primary pages:

- `docs/index.md`: latest generated weekly digest.
- `docs/archive.md`: accepted paper archive plus weekly history.
- `docs/info.md`: reader, submitter, and maintainer guidance.

## Automation

The repository includes:

- `scripts/preview_issue.py`: builds an issue preview comment.
- `scripts/accept_issue.py`: converts an accepted issue into paper YAML.
- `scripts/build_docs.py`: derives weekly digests and generated pages.
- `.github/workflows/issue-curation.yml`: label-driven curation workflow.
- `.github/workflows/validate.yml`: metadata, docs, and tests.
- `.github/workflows/build-site.yml`: MkDocs build.

## Public Boundary

The public website is a convenience layer, not a write API:

- It contains no GitHub token.
- It creates no accepted records.
- It opens a GitHub issue that remains outside the public archive until review.
- Accepted records require maintainer labels and a generated pull request.
- Metadata fetchers reject local, private-network, and non-public URLs before
  fetching content.

## Current Scope

Implemented:

- URL-first GitHub issue template.
- Static website Submit form that opens prefilled GitHub issues.
- Best-effort metadata preview and acceptance scripts.
- Local/private URL rejection for issue automation.
- Label-driven curation workflow that opens pull requests.
- `accepted_at`-derived weekly digest generation.
- Optional weekly override files.
- Generated README, latest weekly page, archive, and info page.

Not implemented:

- Fully reliable metadata extraction for every publisher URL.
- Automatic relevance scoring.
- Email newsletter or RSS distribution.
- Database-backed search.
